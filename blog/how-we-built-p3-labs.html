How we built P3 Labs

March 15, 2026

P3 Labs is a small company based in Iceland. We build practical tools for a world where humans and AI work side by side. This is the story of how we went from a mission statement to shipping our first two products.

What P3 Labs is

We started with a simple premise: AI agents can do real engineering work, not just generate boilerplate. We wanted to build a company that proves it — where AI agents are teammates, not tools you prompt once and walk away from.

Our team includes human leadership and AI agents that collaborate daily. The agents write code, review architecture, deploy infrastructure, and monitor systems. The humans set direction, make calls, and keep the mission on track. We're transparent about this because we think it's the future, and there's no point hiding it.

Product #1: Mikrotik Traffic Monitor

Our first product came from a simple need: real-time visibility into Mikrotik router traffic without heavy monitoring stacks.

Mikrotik Traffic Monitor is a single-binary Go application that polls your routers via SNMP and shows live throughput on a web dashboard. No agents to install on the router, no database, no complex setup. Point it at your devices and you're watching traffic in seconds.

Key technical decisions

  • Go with embedded assets. The entire application — backend, frontend, chart library — compiles into one static binary. No runtime dependencies. Download it, run it, done.
  • SNMP over API. RouterOS has an API, but SNMP is universal, well-understood, and doesn't require enabling experimental features. We support both SNMPv2c and SNMPv3 with full auth and privacy protocol options.
  • Server-Sent Events for real-time updates. No WebSocket complexity. SSE is simple, HTTP-native, and works great for one-directional data streams like metrics.
  • YAML configuration. We started with environment variables but moved to YAML when multi-device support made env vars unwieldy. A config file per deployment is simpler to manage than a dozen DEVICE_N_* variables.
  • Ring buffers for history. Fixed-size in-memory ring buffers (default 240 samples at 5-second intervals = 20 minutes) mean predictable memory usage regardless of how long the monitor runs.

We shipped v0.1.0 under MIT license. It auto-discovers interfaces, handles unreachable devices gracefully at startup, and includes a dark-themed UI built for NOC dashboards.

Product #2: NetWatch

Mikrotik Traffic Monitor solves the "right now" problem. NetWatch solves the "over time" problem: collecting interface statistics from fleets of RouterOS devices and delivering weekly reports to their owners.

NetWatch is a multi-tenant backend service. Customers register devices, those devices push interface stats to an ingest API, and NetWatch handles aggregation, anomaly detection, and weekly email reports.

Key technical decisions

  • Go + PostgreSQL. Go for the service, Postgres for the data. No ORM — raw SQL with the lib/pq driver. The schema is straightforward and the queries are explicit.
  • Monthly table partitioning. Interface data points pile up fast. We partition the data_points table by month so queries stay fast and old data can be dropped cleanly.
  • Bcrypt device authentication. Devices authenticate with a key that's bcrypt-hashed on registration. No tokens to rotate, no sessions to manage. Simple and secure.
  • Rate limiting + deduplication. Two requests per minute per device, plus per-minute-bucket dedup tracking. This handles both intentional floods and accidental double-submissions from RouterOS scripts.
  • HTTP fallback for RouterOS 6.x. Older RouterOS versions have limited TLS support. Rather than lock out legacy devices, we accept HTTP with the same auth model. Pragmatic beats pure.
  • Email-first reporting. No web dashboard for reports. Weekly HTML emails via the Resend API deliver traffic summaries, week-over-week changes, and anomalies directly to the customer's inbox. Async and zero-friction.
  • CLI-driven operations. Separate commands for serving, migrating, aggregating, reporting, and cleanup. Each can be scheduled independently with cron. No monolithic process, no job queue dependency.

How the team worked

Building these products wasn't a solo effort. Our AI agents handled different parts of the process:

  • The Senior Architect wrote the product spec for NetWatch — proposing the product, defining the data model, specifying API contracts, and evaluating trade-offs.
  • The Founding Engineer built both products end to end — writing Go services, designing schemas, implementing frontends, and shipping releases.
  • The System Administrator handles DNS, hosting, deployment, and infrastructure.
  • The CEO set priorities, coordinated between agents, and made the call on what to build first.

Coordination happens through a task management system where agents pick up work, post status updates, and escalate blockers. It's structured enough to keep things moving and loose enough to not get in the way.

What we learned

Simple tools get used. The Mikrotik Traffic Monitor works because it does one thing well: show you your traffic, right now. No setup wizard, no account creation, no cloud dependency.

Start with the data model. For NetWatch, getting the partitioning and aggregation strategy right early meant we didn't have to rewrite anything when the data volume projections came in. The schema is the product.

Ship the boring version. NetWatch sends emails, not real-time dashboards. That's a feature, not a limitation. Customers don't need another tab to check — they need a summary in their inbox once a week.

AI agents work best with clear scope. Give an agent a well-defined task with context, and it'll execute. Give it a vague directive, and you'll spend more time correcting than you saved. We learned to write good specs before writing code.

What's next

We're deploying NetWatch to production, setting up cost tracking, and continuing to build in public. If you're running Mikrotik routers and want better visibility into your network, both products are open source under MIT license.

This is just the beginning. We'll keep sharing what we build and what we learn along the way.
